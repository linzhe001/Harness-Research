# Complete Guide to the CV Research Workflow

This document provides a detailed introduction to the CV research workflow system built around Codex Skills.
The system moves a CV research project through the full path from "idea" to "competition submission / paper-ready state," with automated management and quality control at each step.

---

## 1. System Architecture Overview

### 1.1 Three-Layer Configuration System

The system uses Codex's three-layer configuration mechanism, organized by load frequency and scope of effect:

| Layer | Location | Loading Method | Purpose |
|------|------|---------|------|
| **AGENTS.md / CLAUDE.md** | Project root | Read per session | Global context: project name, environment, tech stack, current stage |
| **Shared refs** | `.agents/references/` | Loaded on demand by skills | Behavioral constraints: code modification conventions, pre-training git operations, dependency-change reminders |
| **Skills** | `.agents/skills/` | Invoked by the user through `$skill-name` or natural language | Stage executors: the full logic for each workflow stage |

**Design philosophy**: AGENTS.md / CLAUDE.md stay minimal and contain only stable information that is needed every time; shared references are loaded on demand to avoid polluting context with irrelevant information; skills are invoked on demand to perform concrete work.

### 1.2 State Ownership

Each state file has a single designated writer to avoid divergence from multiple sources:

| File | Sole Writer | Purpose |
|------|-----------|------|
| `PROJECT_STATE.json` | orchestrator + each WF skill | Stage transitions (single source of truth for workflow stage) |
| `iteration_log.json` | iterate skill | Experiment history (single source of truth for experiments) |
| `project_map.json` | generated by build-plan, maintained by code-debug | Code structure (single source of truth for architecture, stable files only) |
| `CLAUDE.md` | generated incrementally by init-project | Project compatibility snapshot and context summary |

**Key rule**: iterate does **not write** PROJECT_STATE.json; orchestrator does **not write** iteration_log.json. When cross-file information is needed, it is obtained by **reading**.
These canonical state files should all live at the repository root, not inside `.agents/`.

## 2. Orchestrator — Central Dispatcher

**Invocation**: `$orchestrator [init|status|next|rollback|decision]`

Orchestrator does not perform the research work itself. It manages state transitions across the full workflow:

- **`init`** — Initialize the project: create the directory structure, generate PROJECT_STATE.json, and call `$init-project init` to generate a minimal CLAUDE.md
- **`status`** — View current progress: includes stage consistency checks; during WF8 it additionally reads iteration_log.json
- **`next`** — Advance to the next stage: validate prerequisites (whether artifacts are complete and whether blockers exist), then invoke the corresponding skill
- **`rollback`** — Roll back to a specified stage: preserve history and do not delete any artifacts
- **`decision`** — Record a key decision: content, rationale, and considered alternatives

**Automatic trigger**: after a stage is completed, the `next` command automatically calls `$init-project update` to refresh CLAUDE.md (after WF1/WF2/WF4/WF5/WF6).

## 3. Stage Details

### WF1-WF4: Survey → Architecture → Validation → Data

| Stage | Skill | Output | Decision |
|------|-------|------|------|
| WF1 survey-idea | `$survey-idea` | docs/Feasibility_Report.md | PROCEED/PIVOT/ABANDON |
| WF2 refine-arch | `$refine-arch` | docs/Technical_Spec.md | — |
| WF3 deep-check | `$deep-check` | docs/Sanity_Check_Log.md | GO/CONDITIONAL GO/NO-GO |
| WF4 data-prep | `$data-prep` | docs/Dataset_Stats.md + data pipeline + CLAUDE.md dataset path sync | — |

### WF5: Baseline Reproduction (with Mandatory Gating)

| | |
|---|---|
| **Skill** | `$baseline-repro [baseline_name or 'all']` |
| **Output** | `docs/Baseline_Report.md` + baseline_metrics + evaluation_protocol |
| **Gate** | Baseline_Report.md must exist, and each baseline status must be verified/partial (not untested) |

If some baselines are intentionally skipped, they must be marked as `partial` and the reason must be documented in the report.
WF5 also creates the first runnable environment and synchronizes the `## Environment` section of `CLAUDE.md` together with the baseline summary.

### WF6-WF7: Planning → Coding

| Stage | Skill | Output |
|------|-------|------|
| WF6 build-plan | `$build-plan` | docs/Implementation_Roadmap.md + project_map.json |
| WF7 code-expert | `$code-expert` | Full project codebase |

### WF7.5: Code Review + Training Pipeline Validation (Gated)

| | |
|---|---|
| **Skill** | `$validate-run [config_path]` |
| **Review items** | Codex code review (equivalence between new code and the baseline: data pipeline, model, loss, evaluation metrics, common ML bugs) |
| **Validation items** | 100-step training, checkpoint save/load, eval flow, wandb connection, git_snapshot |
| **Gate** | PASS → WF8, REVIEW → continue or fix after user confirmation, FAIL → repair with `$code-debug` |

This ensures that code correctness and infrastructure issues are caught before the iteration stage.

### WF8: Structured Experiment Iteration (Core)

| | |
|---|---|
| **Skill** | `$iterate [plan|code|run|eval|ablate|status|log]` |
| **Output** | iteration_log.json (continuously updated), best checkpoint |
| **Decision** | CONTINUE → WF9 / DEBUG → new iteration / PIVOT → WF2 / ABORT |

**Seven subcommands**:

| Subcommand | Purpose | Utility Invoked |
|--------|------|---------------|
| `plan [hypothesis]` | Record the hypothesis, check repeated lessons, design changes, optionally request a Codex review | — |
| `code [description]` | Implement code changes, mandatory git commit | `$code-debug` |
| `run [config_path]` | Execute training + run eval + collect metrics (automated) | — |
| `eval [log_path]` | Evaluate results, compare with baseline + best, make a decision | `$evaluate` |
| `ablate [iter_id] --components "..."` | Run within-iteration ablations to determine the contribution of each component | — |
| `status` | View the current iteration + latest 5 + best | — |
| `log` | Full iteration history table | — |

**Typical iteration loop**:
```
$iterate plan "Upgrade the backbone from ResNet-50 to ResNet-101 to improve feature representation"
  → Check repeated lessons (warn about known failure patterns)
  → Record the hypothesis and design config_diff
  → (Optional) Codex review of the plan

$iterate code "Upgrade backbone to ResNet-101"
  → Write persistent context (.agents/state/iterations/iter{N}/context.json)
  → Invoke $code-debug to modify code + enforce a git commit
  → Remove the symlink (while keeping the persistent context)
  → status becomes "training"

$iterate run {config_path}
  → Execute training automatically (background task)
  → Automatically run {EVAL_SCRIPT} after training completes
  → Parse stdout to extract metrics and update iteration_log.json
  → status becomes "running" and output a metrics summary

$iterate eval experiments/{exp_prefix}_iter27/
  → Invoke $evaluate to parse metrics → per-iteration report (docs/iterations/iter27.md)
  → Compare with baseline + best and make a decision
  → Output the recommended next command

# Optional: ablation experiment (to determine component contribution)
$iterate ablate iter27 --components "aux_loss:loss.lambda_aux=0.0,lr_warmup:train.warmup_steps=0"
  → Run training w/o each component
  → Output a comparison table (component / primary_metric / Delta / Contribution)
```

**Automated `run` flow**:

The `run` subcommand automatically completes the full training pipeline and replaces the previous manual training workflow:

```
Build training command → Bash(run_in_background) → Parse stdout metrics → Run {EVAL_SCRIPT} → Update iteration_log.json
```

- Automatically build the `python {TRAIN_SCRIPT} --config ... --no_snapshot` command from config_diff
- Use `run_in_background: true` to support long training runs of 10-60 minutes
- After training completes, parse training-trace fields from stdout (such as best step, final step, and intermediate validation summaries)
- Automatically run `{EVAL_SCRIPT}` and extract final metrics according to the evaluation protocol established in WF5
- Error handling: on OOM/NaN/crash, keep `status="training"` and raise an error rather than failing silently
- `--manual` fallback: for cluster training or similar scenarios, fall back to metadata-registration mode

**`ablate` ablation experiment**:

Quickly determine the contribution of each component and generate a comparison table:

```
$iterate ablate {base_iter} --components "name1:override1,name2:override2"
```

Components are passed through `--components` as `name:override` pairs.
For each component, generate a `{base_iter}_no_{component}` sub-iteration, run eval automatically after training, and classify by delta:

| Delta Range | Category |
|-----------|------|
| < -1.0 dB | `significant` — core component |
| < -0.3 dB | `moderate` — contributes meaningfully |
| >= -0.3 dB | `minimal` — can be simplified |
| > 0 dB | `negative` — removal works better |

Supports resume-after-interruption: completed sub-iterations are skipped automatically.

**Other key features**:
- **Persistent context**: stored in `.agents/state/iterations/iter{N}/context.json`
- **Mandatory git commit**: if the code subcommand finishes without a commit hash, remain in `coding` status and do not advance
- **Repeated-lesson check**: during `plan`, scan known lessons and warn about repeated failure modes
- **Screening protocol**: for non-architecture/non-loss changes, a 5K-10K proxy run is recommended first

### WF9: Formal Ablation Experiments

| | |
|---|---|
| **Skill** | `$final-exp [stage_report_path]` |
| **Output** | docs/Final_Experiment_Matrix.md |
| **Prerequisite** | Final WF8 iteration decision is CONTINUE |

Design a complete experiment matrix that meets top-tier conference standards:
- **Ablation experiments**: turn each innovation component ON/OFF to isolate its contribution (can reuse the preliminary results from WF8 `$iterate ablate`)
- **Hyperparameter search**: search space and strategy for key hyperparameters
- **Robustness tests**: different resolutions, extreme scenarios, OOD data
- **Cross-dataset evaluation**: validate generalization
- **Computation budget**: estimate total GPU hours and plan execution order

### WF10: Submission and Release

| | |
|---|---|
| **Skill** | `$release [submit|package|validate]` |
| **Output** | Submission package (multi-scene rendering + packaging + filename validation) |
| **Prerequisite** | WF9 ablation experiments completed |

Three subcommands:
- **`validate`** — Check submission package integrity (filename format, resolution, scene coverage)
- **`package`** — Generate a submission package that meets competition/conference requirements
- **`submit`** — Multi-scene training + rendering + packaging + dry-run check

## 4. Utility Skills

### 4.1 env-setup — Maintenance Environment Refresh

**Invocation**: `$env-setup [create|refresh]`

- Not part of the main workflow prerequisites
- The first runnable environment is created by WF5 `$baseline-repro`
- Use only when dependencies change, the machine changes, or the environment section in `CLAUDE.md` becomes outdated

When `requirements*.txt`, `environment*.yml`, or `pyproject.toml` changes, the `deps-update` rule automatically reminds the user to run `refresh`.

### 4.2 code-debug — Code Repair

**Invocation**: `$code-debug [error_log_path or issue description]`

**Operation modes** (determined by the context in `.agents/state/current_iteration.json`):
- `planned_change`: invoked by `$iterate code`, implements the change according to the hypothesis, then creates a semantic commit
- `bugfix`: invoked independently to diagnose and fix a crash/error
- `perf_tuning`: invoked independently for performance optimization

After modifying code, automatically run `py_compile` + `ruff check`; if interfaces change, also update `project_map.json`.

### 4.3 evaluate — Result Analysis

**Invocation**: `$evaluate [log_path]`

Core functions:
- Parse training logs and extract the target metrics defined by the baseline/evaluation protocol
- Diagnose training issues (overfitting, gradient vanishing, loss divergence, etc.)
- Compare against baseline performance + historical best
- Produce a CONTINUE/DEBUG/PIVOT/ABORT decision

**Per-iteration report**: when invoked from `$iterate eval`, write to `docs/iterations/iter{N}.md`.
`docs/Stage_Report.md` serves as the latest summary index.

## 5. Key Features in Detail

### 5.1 Stable vs Volatile File Layering

project_map.json tracks only **stable architectural files**:
- core modules under `src/`
- subdirectories under `baselines/`
- core configs and scripts (listed in the Entry Scripts section of CLAUDE.md)

**Volatile experimental assets** do not need to be tracked:
- per-iteration scripts (`run_*.sh`, `run_ablation_*.py`)
- temporary experiment configs
- everything under `experiments/`

### 5.2 Pre-Training Git + wandb Integration

Three layers of safeguards ensure that every training run has complete version records:

1. **Claude's semantic commit** (rule-constrained)
2. **git_snapshot.py safety net** (in code)
3. **wandb + checkpoint records** (in code)

### 5.3 Codex Cross-Validation

| Trigger Point | Trigger Condition | Review Target | Review Focus |
|--------|---------|---------|---------|
| WF3 deep-check | **Always triggered** (critical gate) | The technical proposal in Technical_Spec | Find missed risks and failure modes |
| WF7.5 validate-run | **Always triggered** (code entry gate) | New code in `src/` vs reference implementations in `baselines/` | Baseline equivalence: data pipeline, model computation, loss, evaluation metrics |
| WF8 `$iterate plan` | **Triggered selectively**: new loss family, architecture changes, after PIVOT, or 3 consecutive DEBUGs | Single-iteration hypothesis | Validate the hypothesis and avoid repeated failures |

Recorded values: `"used"` / `"skipped_low_value"` / `"unavailable"` (null is no longer used)

### 5.4 Stage-Wise Generation of CLAUDE.md

CLAUDE.md is kept as a **stable operations guide** (≤80 lines) and should not contain fast-changing experiment content.
Fast-changing content (current best, current risk, next experiment) lives in iteration_log.json and MEMORY.md.

| Timing | Content Added |
|------|---------|
| `init` | Environment placeholder + workflow overview |
| After WF1 | Idea description |
| After WF2 | Tech stack details |
| After WF4 | Dataset paths and statistics |
| After WF5 | Baseline metric reference |
| After WF6 | Project Structure + Core Artifacts |
| After the first WF7 experiment | Entry Scripts (locked entry scripts) |

Evaluation reports are stored by iteration:

### 7.3 State File Ownership

```
┌─────────────────────────┐     ┌─────────────────────────┐
│   PROJECT_STATE.json    │     │   iteration_log.json    │
│ single source of truth  │     │ single source of truth  │
│ for workflow stage      │     │ for experiments         │
│                         │     │                         │
│ writers:                │     │ writer:                 │
│   orchestrator          │     │   $iterate skill        │
│   each WF skill         │     │                         │
│                         │     │ contents:               │
│ contents:               │     │   iterations[]          │
│   current_stage         │     │   best_iteration        │
│   history[]             │     │   baseline_metrics      │
│   artifacts             │     │                         │
├─────────────────────────┤     ├─────────────────────────┤
│   project_map.json      │     │   CLAUDE.md             │
│ single source of truth  │     │ global context          │
│ for architecture        │     │                         │
│                         │     │ writer:                 │
│ writers:                │     │   $init-project         │

Key rules: $iterate does not write PROJECT_STATE.json
           orchestrator does not write iteration_log.json
           cross-file information is obtained by "reading"
```

### 7.4 Consistency / Anti-Drift Rules

- Always use one canonical mapping: `WF5=baseline`, `WF6=build-plan`, `WF7=code-expert`, `WF7.5=validate-run`, `WF8=iterate`.
- After WF4 completes, `PROJECT_STATE.json.dataset_paths` and the `### Dataset Paths` section in `CLAUDE.md` must be synchronized immediately.
- After WF5 completes, the `## Environment` section of `CLAUDE.md` must no longer contain placeholders.
- WF8 `run/eval` must use the baseline/evaluation protocol produced in WF5 to decide which metrics to record; training traces and final evaluation metrics must be stored separately.
- After any WF8 subcommand completes, `PROJECT_STATE.json.current_stage.latest_iteration`, `iteration_count`, and `CLAUDE.md Current stage` must be consistent with the latest iteration in `iteration_log.json`.
- An iteration missing a semantic commit, `run_manifest`, or `lessons` must not be marked as completed.

## 8. Quick Reference

**Common commands**:
```
$orchestrator init          # Initialize project
$orchestrator status        # View current status
$orchestrator next          # Advance to the next stage
$orchestrator rollback 2    # Roll back to WF2
$orchestrator decision      # Record a key decision

$baseline-repro all         # Reproduce all baselines
$validate-run               # WF7.5 training pipeline validation

$iterate plan "hypothesis"  # Plan a new iteration (includes repeated-lesson check)
$iterate code "description" # Implement changes (mandatory git commit)
$iterate run config.yaml    # Execute training + collect metrics automatically
$iterate eval path/to/exp   # Evaluate results + make a decision
$iterate ablate {base_iter} --components "name:override,..."  # Ablation experiment
$iterate status             # View iteration progress
$iterate log                # Full iteration history

$code-debug [error info]    # Fix code issues (can be used independently)
$evaluate [log_path]        # Analyze results (can be used independently)
$env-setup refresh          # Refresh environment snapshot after dependency changes

$release validate           # Check submission package integrity
$release package            # Generate submission package
$release submit             # Multi-scene training + packaging
```

**Git branch strategy**: for a solo project, direct development on `master/main` is acceptable. For team collaboration, stage-based branches can be used if needed.

**Commit conventions** (choose the format by scenario):
- Training-related code changes: `train(research): {description}` or `train(baseline/{name}): {description}`
- Workflow docs/config changes: `[WF{n}] {type}: {message}`, where `type = feat / fix / docs / refactor / exp`
