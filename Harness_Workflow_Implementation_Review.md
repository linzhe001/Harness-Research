## 1. 总体实现模型

Harness Research 不是一个单一进程的 workflow engine。它由三类机制叠加:

```text
framework instructions
  .agents/skills + .agents/references + .claude/skills
        |
        v
project-owned workspace files
  PROJECT_STATE.json + docs/** + iteration_log.json + project_map.json
        |
        +--> evidence tooling
        |      init_context / compile_protocol / compile_doc
        |      check_context_gates / check_protocol_drift
        |      check_docchain_gates / check_workflow_state
        |      check_dynamic_context / build_review_packet
        |
        +--> WF10 auto-iterate controller
               .auto_iterate/state.json + lock + events
               Codex phase prompts -> $iterate writes iteration_log.json
```

专家检查点:

- [ ] workflow 阶段本身主要由 skill 文件和共享规则定义，而不是由一个统一 Python 状态机执行。
- [ ] 可执行 Python 工具主要覆盖 dynamic context gate、docchain gate、workflow-state consistency 和 WF10 auto-iterate。
- [ ] 具体研究项目的事实、合同、迭代记录和运行状态属于 project-owned 文件，不属于 framework 源树。

来源: `README.md`, `.agents/references/workflow-guide.md`,
`.agents/skills/orchestrator/SKILL.md`, `tooling/evidence/*.py`,
`tooling/auto_iterate/scripts/auto_iterate/controller.py`。

## 2. Canonical Workflow

当前 canonical 编号在 `README.md` 和 `.agents/references/workflow-guide.md`
中是一致的:

```text
WF1  survey-idea
  -> WF2  idea-debate
  -> WF3  refine-idea
  -> WF4  data-prep
  -> WF5  baseline-repro
  -> WF6  refine-arch / architecture-design
  -> WF7  build-plan
  -> WF8  code-expert
  -> WF9  validate-run
  -> WF10 iterate
  -> WF11 final-exp
  -> WF12 release
```

WF10 内部循环:

```text
plan -> code -> run -> eval
                   |
                   v
      NEXT_ROUND / DEBUG / CONTINUE / PIVOT / ABORT
```

决策语义:

| 决策 | 目标状态 |
|---|---|
| `NEXT_ROUND` | 留在 WF10，进入普通下一轮 |
| `DEBUG` | 留在 WF10，进入修复导向下一轮 |
| `CONTINUE` | WF10 结束，可交给 WF11 |
| `PIVOT` | 回滚到 WF2 idea debate/refine idea |
| `ABORT` | 终止项目方向 |

来源: `README.md`, `.agents/references/workflow-guide.md`,
`.agents/skills/iterate/SKILL.md`, `tooling/auto_iterate/scripts/auto_iterate/controller.py`。

## 3. 分层上下文模型

`context-layering-policy.md` 把信息来源分成 8 层:

```text
L1 core invariants          .agents/references/*.md
L2 operator preferences     OPERATOR_CONTEXT.md
L3 local/private inputs     *.local.*, .env.local
L4 research evidence        docs/30_evidence/**
L5 dynamic protocol draft   docs/35_protocol/**
L6 approved contracts       docs/10_contract/**
L7 current facts/docs       docs/20_facts/**, CLAUDE.md
L8 runtime lessons/state    iteration_log.json, docs/40_iterations/**,
                            docs/50_memory/**, MEMORY.md
```

核心约束:

- Operator context 只能影响偏好，不能证明 project facts。
- Research evidence 不能直接升级为 approved contract。
- Approved contracts 覆盖 protocol drafts 和 operator preferences。
- Auto-iteration observations 不能直接进入 `MEMORY.md`。

来源: `.agents/references/context-layering-policy.md`,
`.agents/references/research-invariants.md`,
`OPERATOR_CONTEXT.md.template`, `MEMORY.md.template`。

## 4. 状态所有权

```text
PROJECT_STATE.json        stage flow / milestones
     writer: orchestrator for stage transitions; stage skills may record
             artifact metadata when invoked
     reader: iterate, gates, auto-iterate status

iteration_log.json        experiment source of truth
     writer: $iterate
     reader: orchestrator, evaluate, auto-iterate postconditions

project_map.json          stable implementation map
     writer: generated by $build-plan; updated by $code-expert, $code-debug,
             or any workflow step that changes stable file presence or
             stable interfaces
     reader: code-expert, validate-run, code-debug

.auto_iterate/**          controller runtime state
     writer: auto-iterate controller
     reader: orchestrator/status checks

.evidence/**              evidence chains, review packets, protocol drafts
     writer: evidence tooling invoked by stages
     reader: gates and human reviewers
```

专家检查点:

- [ ] `$iterate` 不应写 stage transition 到 `PROJECT_STATE.json`。
- [ ] Orchestrator 不应写 `iteration_log.json`。
- [ ] 任意稳定文件新增、删除、重命名或稳定接口变化都应同步
  `project_map.json`。
- [ ] 需要写 `.evidence/**` 的 stage 应调用 evidence tooling，不应手写
  `.evidence/**`。
- [ ] Auto-iterate controller 不应写 `iteration_log.json` 或 `PROJECT_STATE.json`; 它通过 Codex runtime prompt 调用 `$iterate`，然后用 postcondition 检查 repository state。
- [ ] `.auto_iterate/` 是 controller-owned，不应由 `$iterate` 写入。

来源: `.agents/references/workflow-guide.md`,
`.agents/skills/orchestrator/SKILL.md`, `.agents/skills/iterate/SKILL.md`,
`tooling/auto_iterate/scripts/auto_iterate/controller.py`,

## 6. Dynamic Context Gate 实现

Dynamic context 的目标是让证据、protocol draft、approved contract 和当前文档之间有可审计链条。

```text
docs/30_evidence/**          evidence tables
        |
        v
compile_protocol.py          draft protocol packet
        |
        v
docs/35_protocol/**          protocol drafts, Status: draft
        |
        v
doc-compiler / compile_doc.py
        |
        v
.evidence/chains/<doc>/<build>/
  evidence_chain.json
  source_manifest.json
  doc_audit.json
        |
        v
check_dynamic_context.py
  context gate
  protocol drift gate
  docchain gate
  workflow state gate
        |
        v
build_review_packet.py -> human approve/revise/reject
```

### 6.1 `init_context.py`

作用:

- 创建 `docs/10_contract`, `docs/20_facts`, `docs/30_evidence`,
  `docs/35_protocol`, `docs/40_iterations`, `docs/50_memory`,
  `.evidence/chains`, `.evidence/schemas`。
- 从 `templates/docs/**` 和 `schemas/**` 拷贝模板。
- 默认不覆盖已有文件。
- 使用 `--set-state` 时设置 `PROJECT_STATE.json.workflow_mode` 和
  `context_model_version`。

来源: `tooling/evidence/init_context.py`, `README.md`,
`AI_AGENT_SETUP.md`。

### 6.2 `compile_protocol.py`

作用:

- 读取 `docs/30_evidence/Paper_Table.md`,
  `Repo_Table.md`, `Dataset_Table.md`, `Baseline_Table.md`,
  `Metric_Table.md`, `Open_Questions.md`。
- 默认写到 `.evidence/protocol_compiler/<build_id>/docs/35_protocol/**`。
- `--apply --overwrite` 才会写到 `docs/35_protocol/**`。
- 生成的 protocol 仍是 draft，并标记 `Review required: yes`。

来源: `tooling/evidence/compile_protocol.py`,
`.agents/skills/protocol-compiler/SKILL.md`。

### 6.3 `compile_doc.py` 和 docchain gate

作用:

- `compile_doc.py` 不负责写正文；它读取已有 Markdown 和显式 source artifacts。
- 它插入或更新 Markdown 顶部的 `Evidence chain`, `Evidence audit`,
  `Audit result` header。
- 它生成 `evidence_chain.json`, `source_manifest.json`, `doc_audit.json`，
  并更新 `.evidence/index.json`。
- `check_docchain_gates.py` 检查当前 docs 下 `docs/10_contract`,
  `docs/20_facts`, `docs/35_protocol` 的 evidence chain 是否存在、结构是否合法、
  Markdown hash 是否 stale。
- Contract docs 还要求更强证据: fact markers、non-Markdown source、
  medium/high-confidence fact、`supports` 或 `partially_supports` relation。
- `.evidence/**` 是 tooling-owned；stage 需要 evidence artifact 时应调用
  `compile_doc.py`, `compile_protocol.py`, `check_dynamic_context.py` 或
  `build_review_packet.py`。
- Python tooling 记录 read set、source hashes、Markdown hash、git context、
  patch/untracked snapshots、fact markers 和 audit checks；它让证据记录可重复，
  但不自动证明语义支持。未传给 `--source` 的资料不会进入完整 read set。

强制调用点:

| Workflow point | Python tool | 后续价值 |
|---|---|---|
| dynamic-context init | `init_context.py --set-state` | 后续 gate 能识别 dynamic context mode |
| evidence table / open question 改动 | `compile_protocol.py` | 形成 reviewable protocol draft，不直接批准 |
| contract/fact/protocol/release doc 改动 | `compile_doc.py --doc ... --source ...` | 后续 docchain gate 可检查 stale/hash/source |
| WF5/WF10/WF11/WF12 readiness | `check_dynamic_context.py --stage ... --review-packet` | 同一 gate result 给 controller/orchestrator 和人类专家 |
| explicit human approval | `approve_contract.py` | 写入 dual approval markers，后续 gate 才承认 approved |
| stage transition/state edit | `check_workflow_state.py` | 检查 stage artifact、state、iteration、project map 一致性 |

产物更新规则:

- `.evidence/chains/<doc>/<build>/` 是 build-id 版本化产物；旧 build 不应手改。
- 当前 Markdown 的 evidence headers 和 `.evidence/index.json` 会被更新到最新 build。
- Markdown/source/fact marker/support relation 变化时，重新运行 `compile_doc.py`。
- evidence tables 或 open questions 变化时，重新运行 `compile_protocol.py`。
- 每次 approval/release/auto-iterate readiness 前，重新运行 `check_dynamic_context.py --review-packet`。
- approved contract 内容若改变，应回到 `draft` 或 `superseded`，重新 compile doc、
  rebuild packet、再由人类批准。

语义污染边界:

- Python 只记录 provenance/hash/marker/review metadata，不自动发明事实、metric
  meaning、claim boundary 或 contract scope。
- `compile_doc.py` 默认 `support_relation=context_only`、`fact_confidence=low`；
  stronger support 必须由审查者显式选择。
- Contract gate 会拒绝只有 context-only evidence 的 contract docchain。
- 如果发现语义错误，修正文档或 source 后重新跑工具；不要手改 JSON 掩盖问题。

来源: `tooling/evidence/compile_doc.py`,
`tooling/evidence/check_docchain_gates.py`,
`.agents/references/evidence-chain-rule.md`,
`schemas/*.schema.json`。

### 6.4 `check_dynamic_context.py`

一条命令组合四个 gate:

```text
check_context_gates.py
  + check_protocol_drift.py
  + check_docchain_gates.py
  + check_workflow_state.py
        |
        v
optional build_review_packet.py
```

支持的高层 stage 是 `status`, `wf5`, `wf10`, `wf11`, `wf12`。

执行结果语义:

- `ok=true` 表示四个子 gate 都没有 error。
- `status` stage 可用于盘点当前 workspace；它不是批准 contract 的动作。
- `--review-packet` 会把同一次 gate 结果写成
  `.evidence/review_packets/<stage>/<build_id>/review_packet.md/json`。
- 可信边界: gate 能验证文件状态、hash/docchain、schema 和 approval markers；
  不能替代人类对 evidence 是否足够支持 contract 的判断。

### 6.5 `approve_contract.py`

作用:

- 在人类专家明确批准某个 contract 后，统一更新两个批准记录:
  contract Markdown 的 `Status: approved` / `Human approved: yes`，以及
  `PROJECT_STATE.json.contracts.<contract>` 的 `status`, `approved_at`,
  `approved_by`, `approval_source`。
- 支持 `project_contract`, `evaluation_contract`, `baseline_contract`,
  `claim_boundary`。
- 该工具不自行判断是否应该批准；它要求调用者提供 `--approved-by` 和
  `--approval-source`，并且批准后仍应重新运行 dynamic-context gate。

典型调用:

```bash
python tooling/evidence/approve_contract.py \
  --workspace-root . \
  --contract evaluation_contract \
  --approved-by "<human reviewer>" \
  --approval-source ".evidence/review_packets/wf10/<build_id>/review_packet.md"
python tooling/evidence/check_dynamic_context.py --workspace-root . --stage wf10 --review-packet
```

来源: `tooling/evidence/approve_contract.py`,
`.agents/references/contract-gating-rule.md`,
`.agents/skills/review-packet/SKILL.md`。

来源: `tooling/evidence/check_dynamic_context.py`,
`tooling/evidence/check_context_gates.py`,
`tooling/evidence/check_protocol_drift.py`,
`tooling/evidence/check_docchain_gates.py`,
`tooling/evidence/check_workflow_state.py`。

## 7. WF10 Auto-Iterate 实现

Auto-iterate controller 是本仓库最接近真实状态机的部分。

```text
docs/auto_iterate_goal.md
        |
        v
auto_iterate_ctl.py start
        |
        v
LoopController.start_loop()
  parse goal
  validate goal
  load policy
  load accounts
  acquire lock
  run dynamic-context preflight when tooling exists
  write .auto_iterate/dynamic_context_preflight.json
  write .auto_iterate/state.json
  start heartbeat
        |
        v
phase loop
  plan -> code -> run_screening -> run_full -> eval
        |
        v
PhaseSupervisor.run_phase()
  build brief JSON
  render Codex prompt
  run codex exec
  write stdout/stderr/result
        |
        v
PostconditionValidator
  inspect iteration_log.json and docs/iterations/<iter>.md
        |
        v
decision transition
  NEXT_ROUND / DEBUG -> next round
  CONTINUE / PIVOT / ABORT -> stop with workflow reason
```

Controller-owned files:

| 文件 | 作用 |
|---|---|
| `.auto_iterate/state.json` | loop runtime state |
| `.auto_iterate/lock.json` | process lock and heartbeat |
| `.auto_iterate/events.jsonl` | append-only event log |
| `.auto_iterate/dynamic_context_preflight.json` | start 前 WF10 dynamic-context gate summary |
| `.auto_iterate/goal.md` | active goal snapshot |
| `.auto_iterate/goal.next.md` | staged next goal |
| `.auto_iterate/runtime/*` | phase brief/result/stdout/stderr |

Postcondition 不是看模型输出文本，而是重新检查 repository:

| phase | postcondition |
|---|---|
| `plan` | exactly one new iteration, `status=planned`, fields present |
| `code` | same iteration `status=training`, has `git_commit`, `git_message` |
| `run_screening` | `screening.status` in `passed/failed/skipped` |
| `run_full` | `full_run.status` in `completed/recoverable_failed/failed`; metrics required when completed |
| `eval` | `status=completed`, valid decision, lessons, metrics, per-iteration report |

来源: `tooling/auto_iterate/scripts/auto_iterate/controller.py`,
`runtime.py`, `postcondition.py`, `goal.py`, `policy.py`, `state.py`,
`events.py`, `accounts.py`,
`tooling/auto_iterate/docs/auto_iterate_goal_template.md`。
