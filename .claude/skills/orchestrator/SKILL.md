# Harness Research Project Orchestrator

Use this skill for workflow status, stage movement, rollback, and durable
project decisions. It owns `PROJECT_STATE.json`; it reads but never writes
`iteration_log.json`.

## Read First

- `PROJECT_STATE.json` and `templates/project-state-schema.json`.
- `iteration_log.json` when current or target stage is WF10.
- `project_map.json` for stable code structure checks.
- Dynamic context rules when present:
  `context-layering-policy.md`, `contract-gating-rule.md`, and
  `evidence-chain-rule.md`.
- Language behavior: `language-policy.md`.

Prefer deterministic evidence commands when available:

```bash
python tooling/evidence/init_context.py --workspace-root . --set-state
python tooling/evidence/check_context_gates.py --workspace-root . --stage wf10-auto
python tooling/evidence/check_dynamic_context.py --workspace-root . --stage wf10 --review-packet
python tooling/evidence/check_protocol_drift.py --workspace-root . --stage wf10
```

to WF11, `PIVOT` rolls back to WF2/WF3, and `ABORT` terminates.

## Commands

### `init`

Ask for project codename, one-sentence idea, target venue/release target,
deadline, optional base codebase path, and primary dataset. Create the standard
workspace skeleton (`docs/`, `.evidence/`, `schemas/`, `src/`, `baselines/`,
`configs/`, `scripts/`, `tests/`, `experiments/`). Create or update
`OPERATOR_CONTEXT.md` only for explicitly stated stable preferences. Invoke
`/init-project init`, then create `PROJECT_STATE.json` with current stage
WF1 `not_started`, empty artifacts/metrics/decisions/history, and the selected
workflow mode.

### `status`

Read `PROJECT_STATE.json`, verify stage ID/name consistency, and report project
name, current stage, completed and remaining stages, latest artifacts,
blockers, and recommended next action. If in WF10, also read
`iteration_log.json`; if `.auto_iterate/state.json` exists, report it read-only.

### `next`

Before advancing, verify the current stage is completed, required artifacts
exist, blockers are resolved, dynamic-context gates have run or are marked
`NOT_RUN`, and the operator explicitly confirms the transition. Do not
auto-advance.

Special gates:
- New projects must complete WF2 and WF3 before WF4-WF6.
- WF4 requires synchronized dataset paths in `PROJECT_STATE.json` and
  `CLAUDE.md`; `AGENTS.md` should point to stable guidance, not duplicate
  volatile paths.
- WF5 requires baseline metrics and verified/partial baseline statuses.
- WF6 must read idea, dataset, baseline, and protocol/contract context.
- WF7 requires `docs/Technical_Spec.md`.
- WF9 must run `/validate-run`; on PASS, run `/auto-iterate-goal` before WF10.
- Unattended WF10 requires an approved Evaluation Contract or explicit operator
  acceptance of a draft, plus surfaced Baseline Contract gaps.
- WF10 to WF11 requires latest completed iteration decision `CONTINUE`.

After a successful stage completion, call `/init-project update` for stages
whose compact guidance must be refreshed. First-time WF8 invokes
`/code-expert all`; WF10 uses `/iterate` subcommands.

### `rollback TARGET_ID`

Set current stage to TARGET_ID with status `in_progress`, append a rollback
record, preserve all history, and do not delete or overwrite artifacts.

### `decision`

Ask for decision, rationale, and alternatives considered. Append to
`PROJECT_STATE.json.decisions` and update timestamps.

## State Rules

- `PROJECT_STATE.json` is the stage source of truth.
- `iteration_log.json` is owned by `/iterate`.
- `project_map.json` is generated by build-plan and maintained by code-writing
  steps that change stable files or interfaces.
- Report Gate Evidence for stage movement. If a gate is skipped or unavailable,
  say `NOT_RUN` with the reason.
- Keep workflow IDs, schema keys, stage names, decisions, and commands stable;
  localize surrounding prose according to language policy.

## Hard Constraints

- Never advance, approve, pivot, or release without explicit operator
  confirmation or auditable approval artifact.
- Never delete artifacts during rollback.
- Never write `iteration_log.json`.
- Always validate stage name consistency before reporting readiness.
